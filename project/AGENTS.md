# Critical Rules

## 構成変更
- システム構成・スキーマ・設定変更を行う場合は、必ず事前にユーザーの承認を得ること
- 影響範囲と変更内容を説明した上で確認を取ること

## 確認方法
- 変更前に以下を提示すること：
  - 実行内容
  - 影響範囲
  - ロールバック方法（可能な場合）


# Strict Constraints

## データベース操作
- データベースへの書き込み（INSERT / UPDATE / DELETE）は、必ず事前にユーザーの確認を取ること
- seed データ挿入は絶対に実行してはいけません。既存のデータを上書き、削除する行為は禁止です。
- 確認なしにデータを変更してはならない
- 外部DB（Turso）を破壊しないことを前提に実装してください。


# Performance / Optimization Policy

## 基本方針
- 実装時は、同じ仕様をより少ない計算・少ない通信・少ない描画で実現できないかを必ず検討すること。
- 速度改善は、まず計測・コード確認・影響範囲の把握を行い、推測だけで大きな構造変更をしないこと。
- 既存仕様を変えずに改善できる箇所を優先すること。UIやデータ構造の意味が変わる最適化は、事前にユーザーへ説明して承認を得ること。
- 読みやすさを犠牲にした過度なマイクロ最適化は避けること。ただし、不要なループ・重複処理・無駄なDB往復・無駄な再レンダーは積極的に削減すること。

## DBアクセス最適化
- 画面で不要なデータは取得しないこと。ページや用途ごとに必要な取得関数を分け、SQL側で `archived`, `parent_id`, 日付範囲などを絞り込むこと。
- N+1クエリ、行ごとの相関サブクエリ、ループ内のDBアクセスは、可能な限りJOIN・集約・一括取得・Map化で置き換えること。
- 外部DB（Turso）への往復回数を減らすこと。複数の関連する書き込みは、可能な限り `batch(..., "write")` などの原子的な一括処理にまとめること。
- 頻繁に検索・絞り込み・JOINに使う列には、必要性を確認した上で `CREATE INDEX IF NOT EXISTS` を使うこと。
- インデックス追加は非破壊だがスキーマ変更に該当するため、対象・目的・影響範囲を説明してから実施すること。

## フロントエンド最適化
- 描画中に `Array.find`, `Array.filter`, `Array.map` を大量に繰り返さないこと。必要に応じて `Map`, `Set`, `useMemo` で事前計算すること。
- Reactコンポーネント内で、入力が変わらない派生データは `useMemo` や専用の事前計算に寄せること。
- リストの各行に重いモーダルや大きなコンポーネントを常時生成しないこと。選択中の1件だけを描画する構成を優先すること。
- 大量行・長いタイムライン・広いグリッドなどは、仮想化、表示範囲の制限、サーバー側フィルタ、初期表示の軽量化を検討すること。
- ドラッグ・ホバー・スクロール中に全体再計算が起こらないよう、状態の粒度と再レンダー範囲を小さく保つこと。

## 破壊的DB変更の禁止
- `DROP`, `TRUNCATE`, テーブル再作成、DB初期化、全テーブル削除、既存データの一括上書きは絶対に行わないこと。
- 既存データを削除する処理を追加・変更する場合は、必ず `WHERE` 条件と対象範囲を明示し、ユーザーの事前承認を得ること。
- `DELETE FROM table` のような無条件削除は原則禁止。既存仕様として必要な全置換処理がある場合でも、対象テーブル・目的・復旧可能性を確認してから扱うこと。
- マイグレーションや初期化処理に、データ消失につながる処理を混ぜないこと。
- seed処理を新規追加・実行しないこと。既存DBが空であることを理由に、ユーザー確認なしでデータを投入しないこと。
- 本番・preview・dev の接続先を確認し、Turso本番DBに対して検証用の書き込みを行わないこと。
- DB安全確認では、少なくとも `DROP`, `TRUNCATE`, `DELETE FROM`, `ALTER TABLE`, `CREATE INDEX`, `executeMultiple`, `batch` の差分を確認すること。

## 検証基準
- 最適化後は `npm run lint` と `npm run build` を実行すること。
- DB処理を変更した場合は、読み取り・追加・編集・削除・アーカイブ・復帰など、影響する主要操作をテストDBで確認すること。
- テスト用に作成したデータは、確認後に削除し、DBに残っていないことを読み取り確認すること。
- 速度改善のために挙動が変わった可能性がある場合は、変更前にできた操作が引き続き実行可能か確認すること。
